## 输入

1. tool-intake 输出的 JSON。
2. db-design-intake 输出的 JSON。
3. api-design-doc 输出的设计文档。
4. 用户当前补充的提示词。

## 目标

输出一份 markdown 测试文档，覆盖前端、后端、接口、联调、越界检查。

## 固定执行顺序

1. 读取工具身份、核心动作、页面状态、API 契约。
2. 将每个核心动作映射为前端测试点、后端测试点、接口测试点。
3. 将每个关键接口映射为成功场景、失败场景、权限场景。
4. 补齐联调检查与越界检查。
5. 先展示草稿给用户审核，再输出最终文档。

## 规则

1. 只基于当前输入生成测试文档，不得擅自补充未被输入支持的功能点。
2. 每个 core_action 必须至少映射到一个测试点。
3. 每个已定义接口必须至少映射到一个接口检查点。
4. 必须写明前端测试路径与后端测试路径。
5. 必须写明至少一条联调主链路。
6. 必须写明越界检查要求。
7. 审核通过前不得输出最终文档。
8. 最终输出时，只输出测试文档，不附加解释文字。

## 固定输出结构

```md
# <tool_key> 测试文档

## 一、工具上下文
- group:
- tool_key:
- frontend_test_root:
- backend_test_root:

## 二、前端测试点
- 路由可访问
- 导航入口可访问
- 核心成功流程
- 关键状态分支

## 三、后端测试点
- 接口成功流程
- 参数校验
- 权限校验
- 关键业务规则

## 四、接口检查点
### 1. <endpoint_id>
- success_case:
- failure_case:
- permission_case:

## 五、联调检查点
- 登录 -> 侧边栏 -> 工具页面 -> 主操作

## 六、越界与回归检查
- allowed_files 检查
- 契约一致性检查
- 共享能力回归检查

## 七、验收门槛
- 前端测试通过
- 后端测试通过
- 主链路通过
```
