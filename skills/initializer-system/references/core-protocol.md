# 初始化核心协议

本协议描述的是“多 Agent 流水线系统”本身，不绑定任何语言或框架。

## 1. 系统目标

目标系统必须满足以下结构：

1. 用户只直接与架构师对话。
2. 架构师负责澄清、拆解、派单、复核、退回修复。
3. 执行角色负责实现或验证单个可交付边界。
4. 程序负责线程、状态、消息、持久化、日志、diff、review 记录。
5. skill 负责约束各角色的行为边界和输出格式。

## 2. 核心业务抽象

初始化后的系统必须把业务工作单元收敛为：

- `group`
- `tool`

`group -> tool` 是产品与任务拆解模型，不是某种语言的专属目录写法。

一个 `tool` 必须是能独立设计、实现、测试、验收的业务单元。

## 3. 固定交付链

架构师对单个 `tool` 的工作链至少包括：

1. 需求澄清
2. 设计文档
3. 测试文档
4. 任务清单
5. 派单包
6. 执行报告
7. 验收结论

如果缺任一项，流水线不可视为闭环。

## 4. 多轮修复闭环

任何执行角色都必须默认存在以下循环：

1. 接任务
2. 产出初版
3. 被架构师或 reviewer 复核
4. 接收 change request
5. 修复并再次汇报

因此 role skill 必须支持多轮往返，而不是一次性问答。

## 5. 角色边界

角色边界必须写清：

1. 读哪些输入物
2. 写哪些文件
3. 不能写哪些文件
4. 何时必须停止并回报缺失输入
5. 必须输出什么报告

执行角色不得假设自己是仓库里唯一在工作的 Agent。

## 6. 项目专属变量

以下内容必须由项目画像提供，而不是写死在核心协议中：

1. 技术栈
2. 真实目录映射
3. 路由接入点
4. 测试命令
5. 数据迁移机制
6. 角色类型
7. role skill 文件路径

## 7. 初始化后的真相源顺序

建议顺序：

1. 用户已确认需求
2. 项目画像
3. generated skills manifest
4. 架构师设计文档
5. 架构师测试文档
6. 架构师任务清单
7. 角色 skill
8. 仓库架构文档
9. 现有代码，仅作兼容性上下文
