# Skill 可优化点记录

## 1. 架构师接收完整需求后应先落地 TODO 文件

当用户一次性发送大量需求笔记时，架构师 skill 应要求先创建一个需求 TODO 文件，把需求拆成可讨论、可确认、可派单的条目。

该 TODO 文件应至少包含：

- 已明确需求
- 待确认问题
- 阻塞数据库设计的问题
- 阻塞 API 设计的问题
- 阻塞前端页面设计的问题
- 可后续派发给 FE、BE、INT 的任务草案

后续开发应围绕该 TODO 文件逐条与用户确认，而不是直接进入编码。

## 2. 启动失败排障应优先判断代码/迁移配置，不应默认删除应用镜像

当 Docker 启动日志出现 Alembic、导入错误、迁移错误时，skill 应要求先定位代码中的导入路径、模型注册、迁移脚本和数据库状态。

例如当前错误：

```text
ImportError: cannot import name 'SQLModel' from 'app.models'
```

该错误首先指向 `backend/app/alembic/env.py` 与 `backend/app/models.py` 的导入契约不一致，不应默认归因于 Docker 镜像缓存。

skill 应增加排障顺序：

1. 读取报错栈顶和最内层异常。
2. 定位异常引用的真实文件。
3. 检查导入对象是否存在。
4. 检查迁移环境是否仍沿用旧模板。
5. 只有在确认镜像内代码与本地代码不一致、依赖层缓存错误或容器构建上下文错误时，才建议重建镜像。

## 3. 任意用户需求都应先拆解为多个可确认需求项

不论用户输入的是一句话、长笔记、截图描述、报错日志，还是混合了产品需求和技术问题，架构师 skill 都应先把内容拆解成多个可确认的需求项。

拆解后应按顺序逐个与用户确认，禁止一次性把所有模糊点混在一起推进。

该规则应适用于：

- 新增功能需求
- 修改现有功能需求
- 报错排障需求
- 设计讨论需求
- 多角色协作需求
- 混合了产品、前端、后端、测试的问题

推荐流程：

1. 先提取用户输入中的所有独立目标。
2. 再把目标拆成可确认的需求项。
3. 按阻塞程度排序。
4. 每次只推进最关键的一个确认点。
5. 用户确认后再进入下一个确认点。
6. 所有关键边界明确后，再进入设计、派单或编码阶段。

## 4. 开发文档应按开发轮次存储

后续开发中，每一轮开发产生的设计文档、测试文档、任务清单、派单记录、前端报告、后端报告、测试报告，应按轮次集中存储，而不是仅按功能分散存储。

当前按功能路径存储文档容易造成同一轮开发的上下文分散，开发者难以还原当轮决策、输入、输出和验收状态。

建议新增轮次目录规则：

- 每一轮开发创建一个稳定目录，例如 `docs/rounds/round-001/`。
- 该目录内保存本轮全部文档。
- 文件名再表达文档类型和涉及工具，例如 `requirements.md`、`design.md`、`tasks.md`、`frontend-report.md`、`backend-report.md`、`qa-report.md`。
- 如果一轮包含多个工具，应在该轮目录下继续按工具建子目录。
- 功能级长期文档可以保留，但应从轮次文档中汇总生成，不能替代轮次文档。

该规则目前只记录为 skill 优化点，暂不修改现有 skill 文件。

## 5. 大需求应拆成多个工具逐个确认和交付

当用户提出一个包含多个能力的大需求时，架构师 skill 应先拆成多个工具或多个可独立交付的工具任务，不应试图一次性确认和实现全部需求。

推荐开发节奏：

1. 先把大需求拆成工具列表。
2. 按依赖关系排出实现顺序。
3. 每次只选择一个工具进入详细需求确认。
4. 当前工具需求确认清楚后，生成前端任务、后端任务、测试任务。
5. 用户将任务转发给对应 AI 实现。
6. 用户回传前端报告、后端报告、测试报告。
7. 架构师审查报告和代码边界。
8. 当前工具验收后，再进入下一个工具。

该节奏允许架构师一边继续和用户澄清后续工具需求，一边审查其他 AI 返回的实现报告和代码变更。

## 6. 常规开发链路不再引入测试 AI

后续常规工具开发中，测试 AI 不参与独立派单。

架构师 skill 应改为只生成前端 AI 与后端 AI 可执行任务，并把测试责任拆给前后端：

- 前端 AI 负责前端实现、前端 Playwright 测试、前端构建校验、前端越界自检、frontend report。
- 后端 AI 负责后端实现、后端 API 测试、数据库迁移校验、后端越界自检、backend report。
- 联调检查由前端 AI 与后端 AI 在各自报告中分别说明已验证项和未验证项。
- 架构师根据前端报告、后端报告和代码 diff 做最终审查。

任务清单不再生成独立 QA 段，默认只生成 FE、BE、INT 三段。

`skills/members/测试skill.md` 应标记为停用或备用，只在用户明确要求独立测试 AI 时才使用。

## 7. 测试环境必须与真实/开发环境隔离

后续 skill 应明确要求：自动化测试不得直接污染用户正在使用的开发数据库或真实业务数据。

本次问题表现为：后端 pytest 使用当前 Docker 开发数据库创建大量随机用户，用户管理页因此显示大量垃圾邮箱，并把真实用户挤出前端列表。

需要优化的规则：

- 后端测试必须使用独立测试数据库，不能复用开发库或真实库。
- 前端 Playwright 测试必须使用独立测试后端或独立测试账号命名空间。
- 测试创建的用户、文件、业务记录必须可识别、可清理。
- 测试 fixture teardown 应清理测试数据，尤其是 `User` 这类全局表。
- 架构师派单时必须要求开发 AI 说明测试数据隔离策略。
- 验收时如果发现测试会写入真实/开发数据，应视为阻塞问题，不允许仅靠“测试通过”验收。

推荐实践：

1. 配置单独 `POSTGRES_DB_TEST` 或测试专用 database/schema。
2. pytest 运行前迁移测试库，运行后清理或直接丢弃测试库。
3. Playwright 使用 `test_` 前缀账号，并在测试结束后删除。
4. 禁止把长期手工测试账号和自动化随机账号混在同一用户列表中。
