# 工具体系共享原则

## 1. 一个工具只对应一个实现边界

1. 一个工具必须有唯一的 tool_key。
2. 一个工具必须属于且只属于一个 group。
3. 一个工具在前端和后端都必须有稳定的实现根目录。
4. 一个工具的业务规则不得散落在多个无关目录中。

## 2. 新增代码必须按 group/tool-key 收敛

当前项目的新工具必须按以下根目录组织：

- frontend/src/tools/<group>/<tool-key>/
- backend/app/modules/<group>/<tool-key>/
- frontend/tests/<group>/<tool-key>/
- backend/tests/<group>/<tool-key>/

历史上存在的单层目录只作为兼容性遗留，不再作为新代码范式。

## 3. 路由层与业务层必须分离

1. 前端 route 文件只负责路由定义、页面装配、页面入口导出。
2. 前端工具目录负责组件、状态、请求、类型、校验。
3. 后端 router 只负责接口入口、依赖注入、请求分发、响应装配。
4. 后端 service 负责业务编排。
5. 后端 repository 或 dao 负责持久化访问。

## 4. 设计文档优先于历史实现

1. 当历史实现与已确认设计冲突时，优先已确认设计。
2. 当历史实现需要兼容保留时，架构师必须在任务包中显式写明兼容点。
3. 未写入任务包的历史行为，不得默认视为必须保留的产品规则。

## 5. 任务必须严格单工具收敛

1. 一个前端实现任务只实现一个工具。
2. 一个后端实现任务只实现一个工具。
3. 一个测试任务只验证一个工具及其直接集成链路。
4. 禁止为“顺手统一风格”扩改无关工具。

## 6. 输出必须可交接

每个角色完成工作后，都必须交付可复制执行的信息：

1. 修改了哪些文件。
2. 新增了哪些文件。
3. 依赖了哪些输入物。
4. 执行了哪些验证。
5. 还剩哪些未完成项或风险点。

## 7. 子智能体必须优先使用仓库真实结构

1. 引用路径必须优先使用当前仓库真实存在的路径。
2. 禁止继续沿用仓库中已被架构文档判定为过渡态的示例路径。
3. 当仓库局部 README 与主架构文档冲突时，必须按主架构文档与当前任务包收敛。
