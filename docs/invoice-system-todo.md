# 发票系统需求 TODO 草案

## 当前分类

- 请求类型：新增工具组 + 多个新增工具。
- group：`finance`
- group 中文名称：财务
- 工具组织方式：侧边栏创建“财务”工具组，下面拆多个独立工具入口。
- 总体业务名称：发票报销管理
- 当前阶段：需求澄清，禁止进入数据库字段冻结、API 契约冻结和编码派单。

## 已记录需求

- 系统角色包含管理员和用户。
- 管理员可以签字。
- 用户可以上传购买记录截图。
- 用户可以上传 PDF 发票。
- 系统需要把发票和购买记录匹配在一起。
- 匹配应支持程序自动猜测，具体是否允许人工确认待定。
- 需要统计页面。
- 统计页面需要显示有多少购买记录没有发票。
- 统计页面需要显示哪些记录需要签字。
- 需要专门用于签字的页面。
- 需要展示没有发票的购买记录。
- 需要展示只有发票但没有购买记录的记录。
- 需要全自动生成报销单 Excel。
- 单据编号需要是短编号，形式类似 `1`、`2`、`3`，不是发票上的超长号码。
- 系统需要支持三种语言：简体中文、英文、繁体中文。
- 需要考虑高速通行费场景。
- OCR、PDF 解析、匹配、Excel 生成必须本地运行。
- 不训练模型。
- 不接入大模型。
- 不接入云 OCR 或第三方 AI 服务。

## 待确认问题

- 用户上传购买记录截图后，需要 OCR 识别金额、日期、商户、订单号等字段；OCR 结果只用于上传表单临时预填，用户可以修改，最终只保存用户确认提交后的字段。
- PDF 发票需要自动解析发票号码、金额、税额、开票日期、购买方、销售方等字段；解析结果只用于上传表单临时预填，用户可以修改，最终只保存用户确认提交后的字段。
- 发票和购买记录的自动匹配规则优先级是什么：金额、日期、商户、订单号、发票号码、人工备注？
- 自动猜测匹配后，是否必须由用户或管理员人工确认？
- 签字是上传签名图片、鼠标手写签名、电子确认按钮，还是生成 Excel/PDF 后盖章？
- 管理员签字后是否允许撤销？
- 报销单 Excel 的模板由谁提供，是否已有固定格式？
- 高速通行费是否属于特殊发票类型，是否需要独立字段或独立匹配规则？
- “数据库部署在哪里”需要确认：使用当前项目 PostgreSQL，还是独立数据库、云数据库、文件数据库？
- 单据编号是否全局递增，还是按月份、用户、报销批次分别递增？
- 用户权限范围：用户只能看自己的记录，还是能看部门记录？
- 管理员权限范围：所有记录，还是按部门/项目授权？

- 单据编号生成规则。
- 多语言是仅前端 UI 多语言，还是业务数据字段也需要多语言。

## 阻塞 API 设计的问题

- 上传文件大小限制。
- 上传文件类型限制。
- 自动解析和自动匹配是同步返回，还是后台任务。
- Excel 生成是立即下载，还是生成后进入文件列表。
- 签字动作的法律/业务含义。

## 阻塞前端设计的问题

- 是否需要一个总览页加多个子页面。
- 签字页面是列表批量签，还是逐单详情签。
- 匹配页面是否需要左右两栏手动拖拽/选择匹配。
- 没有发票、只有发票没有购买记录是否独立入口。

## 已确认编码规则

- 语言字典推荐使用 BCP 47 locale tag。
- 简体中文：`zh-CN`
- 英文：`en-US`
- 繁体中文：`zh-TW`
- 如果未来不限定地区，只表示文字系统，可改用 `zh-Hans` 与 `zh-Hant`。
- 前端翻译 JSON 可以直接使用 `zh-CN`、`en-US`、`zh-TW` 作为字典 key。
- Python 变量名、数据库枚举名如不适合使用连字符，可使用 `zh_cn`、`en_us`、`zh_tw`，但 API 输出仍建议使用标准 tag。
- 购买记录截图 OCR 结果不得作为独立业务数据持久化。
- 后端不需要单独存储 `ocr_raw_text`、`ocr_suggested_amount` 等 OCR 中间结果。
- 购买记录最终字段只来自用户确认后的表单提交值。
- PDF 发票解析结果不得作为独立业务数据持久化。
- 后端不需要单独存储 PDF 解析的中间文本或建议字段。
- 发票最终字段只来自用户确认后的表单提交值。
- OCR/PDF 解析可使用本地预训练 OCR 库或 PDF 文本提取库，但运行期不得依赖外网。
- OCR 模型文件和依赖版本必须固定，优先在 Docker 镜像构建阶段准备好。
- 自动匹配必须使用可解释的本地规则评分，不使用大模型判断。

## `purchase_records` 已确认字典与规则

- 币种需要支持常见币种。
- 币种建议使用 ISO 4217 三位代码，例如 `CNY`、`USD`、`EUR`、`JPY`、`HKD`。
- 大类为固定字典。
- 大类选项：
  - 交通费用
  - 膳食 / 应酬费用
  - 汽车费用
  - 其他项目费用
- 小类为固定字典。
- 小类选项：
  - 自动导航承载车
  - 智能喷漆机器人
  - 钢筋折弯与结扎机器人
  - 生产线车队调度
  - 研发部开销
- 当且仅当大类为 `其他项目费用` 时，小类允许非空。
- 当大类不是 `其他项目费用` 时，小类必须为空。
- 购买记录必须保存原始截图文件。
- 原始截图文件用于后续管理员审核、发票匹配和报销留档。
- 购买记录不需要订单号字段。
- 大类和小类需要支持三语言显示。
- 大类和小类在数据库/API 中必须存稳定编码，不存显示文本。
- 前端根据当前 locale 将编码翻译为 `zh-CN`、`en-US`、`zh-TW` 对应文本。
- 购买记录主状态需要包含：
  - `draft`：草稿
  - `submitted`：已提交
  - `approved`：已签字/已批准
  - `rejected`：已驳回
- 发票匹配状态是并行状态，不属于购买记录主状态。
- 发票匹配状态至少包含：
  - `unmatched`：未匹配发票
  - `matched`：已匹配发票
- `matched` 可以和购买记录主状态并存。
- 购买记录提交后可以撤回提交。
- 购买记录匹配发票后仍允许修改，但修改后是否需要重新确认匹配待确认。
- 暂不增加 `match_needs_review` 状态。
- 已匹配发票的购买记录被修改后，暂不自动打回未匹配或待重新确认。
- 普通用户可以创建、查看、编辑自己的购买记录。
- 管理员可以查看所有购买记录。
- 管理员可以驳回购买记录。
- 普通用户和管理员都可以逻辑删除购买记录。
- 逻辑删除后的购买记录只有执行删除的人自己看得到。
- 逻辑删除后的购买记录保留 30 天。
- 逻辑删除超过 30 天后自动清空。
- `purchase_records` 第一版不单独做回收站工具。
- 购买记录列表需要提供筛选：正常记录 / 已删除记录。
- 已删除购买记录支持恢复。
- 恢复后清空逻辑删除标记，回到删除前状态。
- 购买记录截图文件存储在本地文件系统目录。
- 截图文件不以数据库二进制字段保存。
- 数据库只保存截图文件相对路径、原始文件名、MIME 类型、文件大小。
- 推荐存储目录：`runtime_data/uploads/finance/purchase_records/`。
- `draft` 可编辑。
- `submitted` 后不可直接编辑，但可以撤回为 `draft` 后编辑。
- `rejected` 后可修改并重新提交。
- `approved` 后允许撤回。
- `approved` 撤回后建议回到 `submitted` 状态。
- 只有管理员可以撤回 `approved` 状态。
