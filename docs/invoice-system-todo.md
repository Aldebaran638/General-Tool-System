# 发票系统需求 TODO 草案

## 当前分类

- 请求类型：新增工具组 + 多个新增工具。
- group：`finance`
- group 中文名称：财务
- 工具组织方式：侧边栏创建“财务”工具组，下面拆多个独立工具入口。
- 总体业务名称：发票报销管理
- 当前阶段：需求澄清，禁止进入数据库字段冻结、API 契约冻结和编码派单。

## 已记录需求

- 系统角色包含管理员和用户。
- 管理员可以签字。
- 用户可以上传购买记录截图。
- 用户可以上传 PDF 发票。
- 系统需要把发票和购买记录匹配在一起。
- 匹配应支持程序自动猜测，具体是否允许人工确认待定。
- 需要统计页面。
- 统计页面需要显示有多少购买记录没有发票。
- 统计页面需要显示哪些记录需要签字。
- 需要专门用于签字的页面。
- 需要展示没有发票的购买记录。
- 需要展示只有发票但没有购买记录的记录。
- 需要全自动生成报销单 Excel。
- 单据编号需要是短编号，形式类似 `1`、`2`、`3`，不是发票上的超长号码。
- 系统需要支持三种语言。
- 需要考虑高速通行费场景。

## 待确认问题

- 三种语言具体是哪三种？
- 用户上传购买记录截图后，是否需要 OCR 识别金额、日期、商户、订单号？
- PDF 发票是否需要自动解析发票号码、金额、税额、开票日期、购买方、销售方？
- 发票和购买记录的自动匹配规则优先级是什么：金额、日期、商户、订单号、发票号码、人工备注？
- 自动猜测匹配后，是否必须由用户或管理员人工确认？
- 签字是上传签名图片、鼠标手写签名、电子确认按钮，还是生成 Excel/PDF 后盖章？
- 管理员签字后是否允许撤销？
- 报销单 Excel 的模板由谁提供，是否已有固定格式？
- 高速通行费是否属于特殊发票类型，是否需要独立字段或独立匹配规则？
- “数据库部署在哪里”需要确认：使用当前项目 PostgreSQL，还是独立数据库、云数据库、文件数据库？
- 单据编号是否全局递增，还是按月份、用户、报销批次分别递增？
- 用户权限范围：用户只能看自己的记录，还是能看部门记录？
- 管理员权限范围：所有记录，还是按部门/项目授权？

## 阻塞数据库设计的问题

- 购买记录、发票、匹配关系、签字记录、报销单是否分别建表。
- 文件存储方式：数据库只存元数据，原文件存本地目录、对象存储，还是数据库二进制字段。
- 单据编号生成规则。
- 多语言是仅前端 UI 多语言，还是业务数据字段也需要多语言。

## 阻塞 API 设计的问题

- 上传文件大小限制。
- 上传文件类型限制。
- 自动解析和自动匹配是同步返回，还是后台任务。
- Excel 生成是立即下载，还是生成后进入文件列表。
- 签字动作的法律/业务含义。

## 阻塞前端设计的问题

- 是否需要一个总览页加多个子页面。
- 签字页面是列表批量签，还是逐单详情签。
- 匹配页面是否需要左右两栏手动拖拽/选择匹配。
- 没有发票、只有发票没有购买记录是否独立入口。

## 第一轮工具拆分

- `invoice_dashboard`：统计总览。
- `purchase_records`：购买记录上传与管理。
- `invoice_files`：发票 PDF 上传与管理。
- `invoice_matching`：购买记录与发票匹配确认。
- `signature_approval`：管理员签字。
- `reimbursement_excel`：报销单 Excel 生成。
