# Round 005 Plan - invoice_matching

本轮计划开发：`finance / invoice_matching`，用于把购买记录和发票文件匹配在一起。

当前状态：计划与核心需求已基本确认，可继续补正式需求和派单文档。

## 前置依赖

必须先完成或基本稳定：

- Round 001 `purchase_records`：购买记录。
- Round 002 `invoice_files`：发票文件。
- Round 003 `frontend_i18n`：前端语言切换，至少 i18n 架构稳定。
- Round 004 `backend_test_isolation`：后端测试隔离，避免后续测试继续污染开发库。

## 工具边界

- 工具组：`finance`
- 工具 key：`invoice_matching`
- 建议显示名：`发票匹配`
- 建议路由：`/finance/invoice-matching`
- 建议后端 API 前缀：`/api/v1/finance/invoice-matching`

本轮只做发票与购买记录匹配，不做签字、不做报销 Excel、不做复杂统计总览。

## 核心目标

- 展示未匹配购买记录。
- 展示可匹配发票。
- 程序基于本地规则给出候选匹配。
- 用户或管理员可以确认匹配。
- 支持手动取消匹配。
- 标记异常情况：
  - 购买记录没有发票。
  - 发票没有购买记录。
  - 金额/日期/商户疑似不一致。

## 推荐默认方案

如果用户没有特别指定，建议采用以下第一版规则：

- 一条购买记录最多匹配一张发票。
- 一张发票可以匹配多条购买记录。
- 只有已提交或已批准的购买记录参与匹配，草稿不参与。
- 只有 `confirmed` 发票参与匹配，`draft` 和 `voided` 不参与。
- 程序只给出候选，不自动最终确认。
- 普通用户可以确认/取消自己购买记录与自己发票的匹配。
- 管理员可以查看所有候选和所有已确认匹配，但不能代用户确认或取消匹配。
- 匹配后，如果购买记录或发票关键字段被修改，匹配状态变为 `needs_review`。
- 发票作废后，已有匹配进入 `needs_review`，不自动取消。
- 第一版不做批量确认。
- 已逻辑删除的购买记录或发票恢复后，不自动恢复原匹配关系，重新进入候选。
- 一张发票匹配多条购买记录时，要求已匹配购买记录金额合计接近发票金额，误差 `<= 0.01`。
- 管理员页面需要显示发票已分摊金额和剩余金额。

## 推荐匹配评分

第一版使用可解释的本地规则，不用大模型。

候选评分建议总分 100：

- 金额完全一致：50 分。
- 金额同币种误差 `<= 0.01`：视为金额一致。
- 币种一致：10 分。
- 日期相差 0 天：20 分；1-3 天：10 分；4-7 天：5 分。
- 销售方/商户/订单名称文本相似：15 分。
- 备注或文件名包含相同关键词：5 分。

建议阈值：

- `score >= 80`：强候选。
- `60 <= score < 80`：弱候选。
- `score < 60`：默认不展示，除非用户打开全部候选。

## 数据模型初稿

建议新增匹配关系表：`invoice_match`

字段草案：

- `id`
- `purchase_record_id`
- `invoice_file_id`
- `owner_id`
- `status`
- `score`
- `score_breakdown`
- `confirmed_by_id`
- `confirmed_at`
- `cancelled_by_id`
- `cancelled_at`
- `review_reason`
- `created_at`
- `updated_at`

状态草案：

- `suggested`：系统候选。
- `confirmed`：人工确认。
- `cancelled`：人工取消。
- `needs_review`：已匹配但关键字段变化，需要重新确认。

## 页面草案

建议页面分为三个视图：

- `待匹配`：购买记录在左，候选发票在右。
- `已匹配`：查看已确认匹配关系。
- `异常`：无发票购买记录、孤立发票、弱候选。

核心操作：

- 生成/刷新候选。
- 确认匹配。
- 取消匹配。
- 手动选择发票匹配购买记录。
- 查看购买记录截图。
- 查看发票 PDF。
