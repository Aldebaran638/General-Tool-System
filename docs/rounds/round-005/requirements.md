# Round 005 Requirements - invoice_matching

本轮实现单独工具：`finance / invoice_matching`，用于将购买记录与发票文件匹配。

## 工具边界

- 工具组：`finance`
- 工具 key：`invoice_matching`
- 导航显示名：`发票匹配`
- 前端路由：`/finance/invoice-matching`
- 后端 API 前缀：`/api/v1/finance/invoice-matching`
- 前端目录：`frontend/src/tools/finance/invoice_matching/`
- 后端目录：`backend/app/modules/finance/invoice_matching/`

本轮只做购买记录与发票匹配，不做签字、不做报销 Excel、不做复杂统计总览。

## 已确认规则

- 一条购买记录最多匹配一张发票。
- 一张发票可以匹配多条购买记录。
- 草稿购买记录不参与匹配。
- 参与匹配的购买记录状态：`submitted`、`approved`。
- 只有 `confirmed` 发票参与匹配。
- `draft`、`voided` 发票不参与新匹配。
- 程序只给候选，不自动最终确认。
- 第一版不做批量确认。
- 普通用户可以确认/取消自己的匹配。
- 管理员可以查看所有候选、已匹配、异常。
- 管理员不能代用户确认或取消匹配。
- 金额同币种误差 `<= 0.01` 视为一致。
- 日期最多允许相差 7 天作为候选。
- 匹配后关键字段变化进入 `needs_review`。
- 发票作废后，已有匹配进入 `needs_review`，不自动取消。
- 已逻辑删除的购买记录或发票恢复后，不自动恢复原匹配关系，重新进入候选。
- 一张发票匹配多条购买记录时，已匹配购买记录金额合计必须接近发票金额，误差 `<= 0.01`。
- 管理员页面需要显示发票已分摊金额和剩余金额。

## 匹配状态

匹配关系状态：

- `confirmed`：人工确认匹配。
- `cancelled`：人工取消匹配，保留历史。
- `needs_review`：匹配后关键字段变化、发票作废、关联记录被删除等，需要重新确认。

候选匹配不落库，按接口实时计算。

## 候选评分

第一版使用本地可解释规则，不使用大模型。

候选评分总分 100：

- 金额一致或误差 `<= 0.01`：50 分。
- 币种一致：10 分。
- 日期相差 0 天：20 分。
- 日期相差 1-3 天：10 分。
- 日期相差 4-7 天：5 分。
- 销售方/商户/订单名称文本相似：15 分。
- 备注或文件名包含相同关键词：5 分。

候选等级：

- `score >= 80`：强候选。
- `60 <= score < 80`：弱候选。
- `score < 60`：默认不展示。

## 金额分摊

由于一张发票可以匹配多条购买记录，系统需要计算：

- `allocated_amount`：当前发票已匹配购买记录金额合计。
- `remaining_amount`：发票金额减去已分摊金额。

确认匹配时，如果新增购买记录金额后导致发票超额超过 `0.01`，必须拒绝。

## 权限

普通用户：

- 查看自己的可匹配购买记录、可用发票、候选、已确认匹配。
- 确认自己的购买记录与自己的发票之间的匹配。
- 取消自己的匹配。

管理员：

- 查看所有用户的可匹配数据、候选、已确认匹配、异常。
- 查看发票已分摊金额和剩余金额。
- 不能代用户确认或取消匹配。

## 变更影响

当购买记录或发票关键字段被修改时，相关 `confirmed` 匹配应进入 `needs_review`。

购买记录关键字段：

- `purchase_date`
- `amount`
- `currency`
- `order_name`
- `category`
- `subcategory`
- `note`

发票关键字段：

- `invoice_number`
- `invoice_date`
- `invoice_amount`
- `tax_amount`
- `currency`
- `seller`
- `invoice_type`

本轮允许用服务层显式调用方式实现，不要求通用事件系统。
