# Round 005 Questions - invoice_matching

请按顺序确认以下问题。确认后才能生成正式前后端派单。

## 1. 匹配数量关系

默认建议：

- 一条购买记录最多匹配一张发票。
- 一张发票最多匹配一条购买记录。

已确认：

- 一条购买记录最多匹配一张发票。
- 一张发票可以匹配多条购买记录。

## 2. 可参与匹配的状态

默认建议：

- 购买记录：`submitted`、`approved` 可参与；`draft` 不参与。
- 发票文件：只有 `confirmed` 可参与；`draft`、`voided` 不参与。

已确认：

- 草稿购买记录不参与匹配。
- 只有 `confirmed` 发票参与匹配。

## 3. 确认权限

默认建议：

- 普通用户可以确认/取消自己的匹配。
- 管理员可以查看所有匹配。
- 管理员第一版不代用户确认，避免业务责任不清。

已确认：

- 普通用户可以确认/取消自己的匹配。
- 管理员可以查看所有匹配。
- 管理员第一版不代用户确认或取消匹配。

## 4. 匹配评分

默认建议：

- 金额完全一致最重要。
- 币种必须一致。
- 日期允许 7 天以内作为候选。
- 销售方/商户/订单名称作为辅助文本相似度。

已确认：

- 金额同币种误差 `<= 0.01` 视为一致。
- 日期最多允许相差 7 天。

## 5. 字段变化后的处理

默认建议：

- 已确认匹配后，如果购买记录的金额、币种、日期、大类、订单名称变化，匹配变为 `needs_review`。
- 已确认匹配后，如果发票的金额、币种、日期、销售方、发票号码变化，匹配变为 `needs_review`。
- `needs_review` 不自动取消，但在页面显著提示。

已确认：

- 接受 `needs_review`。

## 6. 删除和作废

默认建议：

- 发票作废后，相关匹配变为 `needs_review` 或 `cancelled`。
- 购买记录或发票逻辑删除后，不在普通匹配列表展示。
- 恢复后可以重新进入候选。

已确认：

- 发票作废时，已有匹配进入 `needs_review`，不自动取消。

## 7. 批量操作

默认建议：

- 第一版不做批量确认。
- 第一版只做逐条确认，降低误匹配风险。

已确认：

- 第一版不做批量确认。

## 8. 补充确认

已确认：

- 已逻辑删除的购买记录或发票恢复后，不自动恢复原匹配关系，重新进入候选。
- 一张发票匹配多条购买记录时，要求已匹配购买记录金额合计接近发票金额，误差 `<= 0.01`。
- 管理员需要看到“发票已分摊金额 / 剩余金额”。

## 9. 结论

核心需求已确认，可以生成正式前后端派单。
