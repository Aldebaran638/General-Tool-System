# 1. 项目目标

建设一个企业内部考试管理平台，作为企业微信工作台中的自建应用使用。

系统需要实现：

1. 企业微信工作台免登录进入系统。
2. 从企业微信同步公司通讯录结构与人员信息。
3. 管理员创建考试、编辑试卷、设置考试规则、分配考试人员。
4. 学员在企业微信手机端完成答题。
5. 系统自动判分、判断是否通过、统计考试完成情况。
6. 通过企业微信向学员发送开考提醒和未完成提醒。

第一版不做完整 LMS，不做复杂培训计划、积分、证书、防作弊监控等功能。

# 2. 企业微信接入要求

## 2.1 应用形态

系统作为企业微信管理后台中的“自建应用”接入。

管理员需要在企业微信后台完成：

1. 创建自建应用。
2. 设置应用可见范围。
3. 获取 CorpID、AgentId、Secret。
4. 配置网页授权可信域名。
5. 配置企业可信 IP。
6. 将系统入口地址配置到工作台应用主页。

## 2.2 关键企业微信 API

编码时以企业微信官方文档为准。当前需要用到的核心接口如下：

### 2.2.1 获取 access_token

接口：

```http
GET https://qyapi.weixin.qq.com/cgi-bin/gettoken?corpid=CORP_ID&corpsecret=SECRET
```

用途：

- 后端获取调用企业微信接口的凭证。
- `access_token` 只能保存在后端，不能返回给前端。
- 需要缓存，不能每次接口调用都重新获取。

官方参考：

```text
https://qiyeweixin.apifox.cn/api-10061194
```

### 2.2.2 OAuth 获取当前访问用户身份

企业微信工作台内打开应用时，不能直接读取客户端内部登录态，必须使用 OAuth 获取 `code`，再用 `code` 换取 `UserId`。

接口：

```http
GET https://qyapi.weixin.qq.com/cgi-bin/user/getuserinfo?access_token=ACCESS_TOKEN&code=CODE
```

用途：

- 根据企业微信 OAuth 回调传入的 `code` 获取当前访问成员的 `UserId`。
- 后端用 `UserId` 建立本系统登录态。

官方参考：

```text
https://qiyeweixin.apifox.cn/api-10061337
```

### 2.2.3 获取部门列表

接口：

```http
GET https://qyapi.weixin.qq.com/cgi-bin/department/list?access_token=ACCESS_TOKEN
```

用途：

- 同步企业微信通讯录部门树。
- 根据 `parentid` 判断中心、部门和更深层级部门。

官方参考：

```text
https://qiyeweixin.apifox.cn/api-10061586
```

### 2.2.4 获取部门成员详情

接口：

```http
GET https://qyapi.weixin.qq.com/cgi-bin/user/list?access_token=ACCESS_TOKEN&department_id=DEPARTMENT_ID&fetch_child=1
```

用途：

- 获取部门下成员详情。
- 同步成员 `userid`、`name`、`department`、`position`、`status`、`main_department` 等字段。

官方参考：

```text
https://qiyeweixin.apifox.cn/api-10061205
```

### 2.2.5 发送应用消息

接口：

```http
POST https://qyapi.weixin.qq.com/cgi-bin/message/send?access_token=ACCESS_TOKEN
```

用途：

- 发送开考提醒。
- 管理员手动提醒未完成学员。

官方参考：

```text
https://qiyeweixin.apifox.cn/api-10061348
```

# 3. 用户角色与权限

## 3.1 角色定义

第一版至少包含 3 类角色：

| 角色 | 说明 | 权限 |
|---|---|---|
| 超级管理员 | 系统最高权限，通常为信息化 / HR / 培训负责人 | 企微配置、通讯录同步、全部考试管理、角色管理 |
| 考试管理员 | 创建和维护考试的人 | 创建考试、编辑试卷、分配学员、查看成绩、发送提醒 |
| 普通学员 | 参加考试的员工 | 查看自己的考试、进入答题、提交试卷、查看结果 |

第二版可扩展：

| 角色 | 说明 |
|---|---|
| 中心管理员 | 只管理本中心的考试与学员 |
| 只读查看者 | 只能查看统计，不可编辑 |

## 3.2 权限实现原则

不要直接把“企业微信管理员”当作本系统管理员。

系统内部需要维护角色关系：

```text
wecom_userid -> system_role
```

即使某人不是企业微信管理员，也可以被授予考试管理员权限。

# 4. 企业微信组织结构同步规则

## 4.1 核心定义

系统中的组织口径如下：

```text
企业根目录
  -> 中心
      -> 部门
          -> 更深层级子部门
```

### 4.1.1 中心

中心 = 企业微信通讯录根目录下的一级部门。

要求：

1. 中心不能硬编码。
2. 必须从企业微信部门树动态读取。
3. 根目录下的直属人员不是部门，不能当作中心。
4. 即使一级部门名称不是“中心”，系统中也统一叫“中心”。

例如图 7 中这些一级目录都属于中心：

```text
内部审计部
董秘办
总经办
财务中心
人力行政中心
采购中心
市场中心
销售中心
创新研发中心
原料生产中心
美道事业部（筹）
维创星
品牌孵化中心
```

### 4.1.2 部门

部门 = 中心下面的一级目录。

例如企微真实结构：

```text
创新研发中心
  -> 法规部
      -> 欧美法规组
          -> 标签审核小组
```

系统展示：

```text
中心：创新研发中心
部门：法规部
```

第三层及更深层级不作为前端独立部门展示，但需要保留原始路径。

### 4.1.3 更深层级归并

如果员工实际挂在三级 / 四级部门下，需要向上归并到中心下一级部门。

例如员工真实路径：

```text
创新研发中心 / 法规部 / 欧美法规组 / 标签审核小组
```

系统归并后：

```text
中心：创新研发中心
部门：法规部
原始路径：创新研发中心 / 法规部 / 欧美法规组 / 标签审核小组
```

按“法规部”添加学员时，需要包含法规部及其所有下级部门里的员工。

## 4.2 多部门员工规则

企业微信中一个成员可能归属于多个部门。

系统规则：

1. 前端展示所有中心。
2. 前端展示所有归并后的部门。
3. 按任一中心筛选，都能筛到该员工。
4. 按任一部门筛选，都能筛到该员工。
5. 人数统计按 `userid` 去重。
6. 考试人员唯一键为 `exam_id + userid`。

## 4.3 根目录直属人员规则

如果员工在企微中至少归属于一个中心 / 部门：

```text
前端正常展示其中心、部门
```

如果员工没有任何中心 / 部门归属，只是挂在企业根目录下：

```text
中心：未归属中心
部门：公司根目录
```

此类人员：

1. 可以通过“特定人员添加”加入考试。
2. 不参与按中心添加。
3. 不参与按部门添加。

## 4.4 企微账号状态

人员状态直接使用企业微信账号状态，不使用“正式 / 试用”等 HR 状态。

映射建议：

| 企微 status | 前端显示 |
|---|---|
| 1 | 已激活 |
| 2 | 已禁用 |
| 4 | 未激活 |
| 5 | 已退出企业 |

前端列名建议使用：

```text
企微账号状态
```

不要叫“人员状态”，避免误解为 HR 用工状态。

# 7. 图 1：考试人员管理页

## 7.1 页面定位

某一场考试内的人员管理页面，用于展示当前考试有哪些学员，以及这些学员是否完成考试。

## 7.2 页面入口

```text
考试管理列表页
  -> 点击某场考试“学员”
  -> 进入考试详情页
  -> 默认打开“人员管理”标签
```

## 7.3 顶部统计

第一版需要：

| 指标 | 说明 |
|---|---|
| 学员人数 | 当前考试固定学员名单人数，按 userid 去重 |
| 已完成人数 | 已通过人数 |
| 未完成人数 | 未通过或未提交人数 |
| 完成率 | 已完成人数 / 学员人数 |

注意：本系统中“完成”按“通过”计算，不是按“提交试卷”计算。

## 7.4 筛选条件

第一版需要：

```text
姓名
中心
部门
企微账号状态
考试完成情况
```

可选：

```text
职务
是否提交
是否通过
```

## 7.5 表格字段

必须展示字段：

| 字段 | 来源 | 说明 |
|---|---|---|
| 姓名 | 企业微信 `name` | 公司内部姓名 |
| 中心 | 企微部门树归并结果 | 可展示多个中心 |
| 部门 | 企微部门树归并结果 | 中心下一级部门，可展示多个 |
| 职务 | 企业微信 `position` | 企微通讯录中显示的职务 |
| 企微账号状态 | 企业微信 `status` | 已激活 / 禁用 / 未激活 / 已退出 |
| 考试完成情况 | 本系统计算 | 已完成 / 未完成 |

建议额外展示：

| 字段 | 说明 |
|---|---|
| 最终分数 | 根据最终成绩规则计算 |
| 是否通过 | 是 / 否 |
| 最后提交时间 | 最近一次提交时间 |
| 考试次数 | 已提交次数 / 允许次数 |
| 操作 | 查看记录、提醒、移除等 |

## 7.6 添加学员方式

第一版支持三种：

```text
按中心添加
按部门添加
按特定人员添加
```

### 7.6.1 按中心添加

选择一个或多个中心，系统加入这些中心下所有员工。

规则：

1. 包含中心下所有部门及更深层级人员。
2. 多部门员工只添加一次。
3. 根目录直属人员不属于任何中心，不会被按中心添加。

### 7.6.2 按部门添加

部门定义为“中心下一级目录”。

选择一个或多个部门，系统加入该部门及其所有更深层级子部门员工。

例如选择：

```text
创新研发中心 / 法规部
```

需要包含：

```text
法规部
法规部 / 欧美法规组
法规部 / 欧美法规组 / 标签审核小组
```

### 7.6.3 按特定人员添加

支持按以下条件搜索：

```text
姓名
userid
中心
部门
职务
```

选中后加入考试。

### 7.6.4 去重规则

同一个人可能通过中心、部门、特定人员重复添加。

系统必须按以下唯一键去重：

```text
exam_id + userid
```

## 7.7 发布考试时固定名单

需求确认：

```text
发布考试时生成固定学员名单。
后续企微人员变化不自动影响已发布考试。
```

实现要求：

发布时将当前人员信息写入考试人员快照，包括：

```text
userid
name_snapshot
center_snapshot
department_snapshot
position_snapshot
wecom_status_snapshot
```

考试发布后：

1. 员工转部门，不影响这场考试中的历史中心 / 部门展示。
2. 员工改职务，不影响历史职务快照。
3. 新员工加入企微，不会自动加入已发布考试。
4. 离职员工仍保留历史考试记录。

```text

```

## 8.5 右侧题目设置区

每道题需要配置：

| 字段 | 说明 |
|---|---|
| 正确答案 | 单选 1 个，多选多个，判断 1 个 |
| 分数 | 必填，必须大于 0 |
| 难度 | 低 / 中 / 高，可第一版保留默认值 |
| 考核点 | 可选，第一版可不强制 |
| 答案解析 | 可选，第一版可不做或仅预留字段 |

## 8.6 单选题规则

1. 至少 2 个选项。
2. 最多建议 10 个选项。
3. 只能设置 1 个正确答案。
4. 选项自动编号 A、B、C、D……
5. 支持添加、删除、编辑、排序选项。

## 8.7 多选题规则

1. 至少 2 个选项。
2. 正确答案可以多个。
3. 至少选择 2 个正确答案才是合法多选题。
4. 第一版不做漏选得分。
5. 多选题必须完全一致才得分。

## 8.8 判断题规则

固定两个选项：

```text
A. 正确
B. 错误
```

只能选择其中一个为正确答案。

## 8.9 题目排序

必须支持题目排序。

可用方式：

```text
拖拽排序
上移 / 下移按钮
```

保存时每题写入：

```text
sort_no
```

## 8.10 试卷级信息

试卷编辑器顶部需要展示：

```text
总题数
总分数
保存状态
预览按钮
```

## 8.11 发布前校验

发布考试前必须校验：

1. 试卷至少有 1 道题。
2. 每道题必须有题干。
3. 每道题必须有分数，且分数大于 0。
4. 每道题必须有正确答案。
5. 单选题只能有 1 个正确答案。
6. 多选题至少 2 个正确答案。
7. 判断题必须从“正确 / 错误”中选一个。
8. 试卷总分必须大于 0。

## 8.12 试卷快照

强烈要求：考试发布时生成试卷快照。

原因：

```text
考试发布后，管理员继续修改题目，不应该影响已经开始或已经提交的考试记录。
```

实现建议：

```text
exam_paper_snapshot
exam_question_snapshot
exam_question_option_snapshot
```

学员答题和判分都基于快照，而不是基于实时编辑中的题库数据。

## 9.4 最终成绩规则

需求确认后的最终规则：

```text
如果员工至少有一次通过记录：
    最终结果取最后一次通过记录

如果员工从未通过：
    最终结果取最后一次提交记录
```

示例 1：

```text
第 1 次：60 分，未通过
第 2 次：70 分，未通过
第 3 次：75 分，未通过

最终分数：75
最终通过：否
完成情况：未完成
```

示例 2：

```text
第 1 次：85 分，通过
第 2 次：70 分，未通过
第 3 次：75 分，未通过

最终分数：85
最终通过：是
完成情况：已完成
```

示例 3：

```text
第 1 次：85 分，通过
第 2 次：90 分，通过
第 3 次：70 分，未通过

最终分数：90
最终通过：是
完成情况：已完成
```

实现伪代码：

```pseudo
attempts = 查询该员工该考试所有已提交记录，按 submitted_at 升序
passed_attempts = attempts where passed = true

if passed_attempts 不为空:
    final_attempt = passed_attempts 中 submitted_at 最新的一条
    final_passed = true
    completion_status = COMPLETED
else if attempts 不为空:
    final_attempt = attempts 中 submitted_at 最新的一条
    final_passed = false
    completion_status = NOT_COMPLETED
else:
    final_attempt = null
    final_passed = false
    completion_status = NOT_COMPLETED
```

## 12.4 答题倒计时

第一版防作弊 / 考试策略只做：

```text
答题倒计时
随机题目顺序
随机选项顺序
```

倒计时规则：

1. 学员点击“开始考试”后创建一次考试尝试记录。
2. 根据考试时长计算结束时间。
3. 前端显示倒计时。
4. 时间到后自动交卷。
5. 后端必须校验时间，不能只依赖前端。

## 12.5 随机题目顺序

如果考试设置开启随机题目顺序：

1. 学员第一次开始某次考试时，后端生成题目顺序快照。
2. 快照保存到 `exam_attempt` 或单独顺序表。
3. 前端按快照展示。
4. 刷新页面后顺序不变。

## 12.6 随机选项顺序

如果考试设置开启随机选项顺序：

1. 学员第一次开始某次考试时，后端为每道题生成选项顺序快照。
2. 前端按快照展示。
3. 判分时不能依赖 A/B/C/D 字母，需要依赖 option_id。
4. 学员刷新页面后选项顺序不变。

## 12.7 提交流程

```text
学员点击提交
  -> 前端检查必答题是否已答
  -> 后端再次校验交卷要求
  -> 保存答案
  -> 自动判分
  -> 生成 / 更新考试尝试记录
  -> 计算最终成绩与完成状态
  -> 返回结果页
```

## 12.8 自动判分规则

第一版：

| 题型 | 判分规则 |
|---|---|
| 单选题 | 选项完全一致得满分，否则 0 分 |
| 判断题 | 答案一致得满分，否则 0 分 |
| 多选题 | 必须完全一致得满分，否则 0 分 |

第一版不做：

```text
多选题漏选得分
错选扣分
主观题人工批改
```

# 13. 考试状态流转

建议考试状态：

```text
DRAFT       草稿
PUBLISHED   已发布
NOT_STARTED 未开始
IN_PROGRESS 进行中
ENDED       已结束
ARCHIVED    已归档
```

可简化实现：

```text
数据库存：DRAFT / PUBLISHED / ARCHIVED
前端根据时间计算：未开始 / 进行中 / 已结束
```

规则：

1. 未发布考试可以自由编辑。
2. 发布考试前必须校验试卷、人员、考试设置。
3. 发布时生成试卷快照和人员快照。
4. 已发布考试不建议直接删除，只允许归档。
5. 已发布考试如果允许修改，需要明确是否影响学员端；第一版建议尽量禁止修改核心项。

# 16. 发布考试校验规则


## 17.1 倒计时后端校验

```text

```

提交时判断：

```pseudo
if now > expire_at:
    自动交卷或拒绝继续答题
```

## 17.2 随机顺序快照

不能只在前端随机。

必须：

```text

```

否则刷新页面、重复进入、自动交卷时会出现判分混乱。

# 21. 编码 AI 实现提示

如果将本文档交给本地编码 AI，建议按以下优先级生成代码：

1. 先生成数据库模型和枚举。
2. 再生成企业微信服务层：token、OAuth、部门同步、成员同步、消息发送。
3. 再生成考试管理 CRUD。
4. 再生成人员管理与人员快照逻辑。
5. 再生成试卷编辑与发布快照逻辑。
6. 再生成学员答题、随机顺序快照、自动判分。
7. 最后生成前端页面。

核心注意事项：

```text
1. 不要把企业微信 access_token 暴露给前端。
2. 中心和部门不能硬编码，必须从企微部门树计算。
3. 人数统计必须按 userid 去重。
4. 发布考试时必须固化人员快照和试卷快照。
5. 完成情况按“是否通过”计算。
6. 最终成绩不是简单最后一次，而是：有通过则取最后一次通过；从未通过则取最后一次提交。
7. 随机题目和随机选项必须由后端生成并保存快照。
8. 学员端必须移动端适配。
9. 管理端优先 PC。
```

# 22. 当前已确认需求汇总

```text
1. 所有企微根目录下一级部门都叫中心，但不能硬编码，必须从企微读取。
2. 根目录直属领导不是中心；若无部门归属，显示未归属中心 / 公司根目录。
3. 部门定义为中心下一级目录。
4. 员工挂在更深层级部门时，归并到中心下一级部门。
5. 多部门员工展示所有中心和部门。
6. 人员状态直接用企微账号状态。
7. 学员添加支持按中心、按部门、按特定人员。
8. 发布考试时生成固定学员名单，后续企微变化不自动影响已发布考试。
9. 考试最终结果：有通过则取最后一次通过；从未通过则取最后一次提交。
10. 完成情况按通过计算。
11. 提交后显示分数和是否通过。
12. 是否显示正确答案由管理员配置。
13. 开考前自动提醒一次。
14. 管理员可手动提醒未完成学员。
15. 管理员端优先 PC。
16. 学员端必须适配企业微信手机端。
17. 第一版防作弊只做倒计时、随机题目顺序、随机选项顺序。
```
