# 本地启动与排障说明

这份文档只记录已经实际验证可以跑通的本地启动流程。

目标不是覆盖所有理论路径，而是给出一条当前仓库下可以直接执行、可以自检、遇到常见问题可以处理的路径。

## 1. 适用范围

- 适用于当前仓库的本地开发和联调启动。
- 适用于使用 Docker 和 Docker Compose 启动整套前后端、数据库、代理和辅助工具。
- 不适用于生产部署。生产相关内容仍参考根目录的 deployment.md，但那份文档是模板化说明，不代表当前仓库已经逐项验证。

## 2. 已验证可走通的启动结论

当前仓库本地可走通的方式是：

1. 保证根目录存在 .env 文件。
2. 使用 docker compose watch 启动。
3. 如果本机 5432 已被占用，使用当前仓库里的开发覆盖配置，数据库会自动映射到宿主机 15432，不需要你再手动改一次。

当前仓库已经验证可访问的本地地址：

- 前端：http://localhost:15173
- 后端：http://localhost:8000
- Swagger：http://localhost:8000/docs
- Adminer：http://localhost:18081
- MailCatcher：http://localhost:1180
- MailCatcher SMTP（宿主机）：localhost:11025
- Traefik UI：http://localhost:18090
- PostgreSQL 宿主机端口：15432

## 3. 启动前提

本地至少需要：

- Docker
- docker compose

可以先检查：

```bash
docker --version
docker compose version
```

## 4. 第一次启动

### 第一步：准备 .env

当前项目不能只依赖 .env-example。

原因是 compose.yml 里多个服务使用了：

```yaml
env_file:
  - .env
```

这意味着即使你显式传入其它 env 文件，根目录缺少 .env 仍然会导致 compose 失败。

最简单的做法：

```bash
cp .env-example .env
```

如果你已经有 .env，就不要覆盖。

### 第二步：启动

在项目根目录执行：

```bash
docker compose watch
```

第一次启动会构建前后端镜像，耗时比后续启动明显更长。

## 5. 启动成功的判断方法

不要只看容器是否创建成功，要看服务是否真的可访问。

### 查看容器状态

```bash
docker compose ps -a
```

可接受的状态是：

- backend 为 Up 且 health 为 healthy
- db 为 Up 且 health 为 healthy
- frontend 为 Up
- proxy 为 Up
- mailcatcher 为 Up
- adminer 为 Up
- prestart 为 Exited (0)

注意：

- prestart 退出不是失败。
- prestart 的职责是执行初始化，正常结束时退出码应该是 0。

### 验证后端

```bash
curl http://localhost:8000/api/v1/utils/health-check/
```

期望返回：

```json
true
```

### 验证前端

```bash
curl http://localhost:15173
```

返回 HTML 即表示前端服务已启动。

## 6. 这条路径为什么现在能走通

本仓库已经做过以下修正：

### 6.1 .env-example 已补全

当前 .env-example 不再是空白文件，已经补齐了当前 compose.yml 和后端配置启动所需的核心环境变量。

### 6.2 本机 5432 被占用时不会再卡死

当前 compose.override.yml 已把数据库端口映射改成：

```yaml
15432:5432
```

所以即使你本机已经有 PostgreSQL 占用了 5432，这个项目也仍然可以启动。

### 6.3 docker compose watch 不会再因为没有 watch 配置而直接退出

当前 compose.override.yml 已补了最小的 develop.watch 配置。

之前的失败原因是：

```text
none of the selected services is configured for watch
```

现在这个问题已经被消除。

## 7. 当前仍然存在但不阻塞启动的问题

### 7.1 compose watch 会提示部分路径不会被监控

当前 watch 运行时会出现类似告警：

```text
path '.../backend/app' also declared by a bind mount volume, this path won't be monitored
```

这是因为这些目录同时被 bind mount 挂载了。

这个告警不影响项目跑通，原因是：

- 后端本身已经用 fastapi run --reload
- 前端本身已经用 Vite dev server

也就是说，本地开发时热更新仍然是有效的，只是不是靠 compose watch 的文件同步实现。

### 7.2 Traefik 会提示 certificate resolver 不存在

本地代理日志里会出现 le resolver 相关警告。

这是本地开发环境没有配置真正的 Let's Encrypt 解析器导致的。

这个问题不会影响你通过 localhost 端口访问前端、后端、Adminer 和 MailCatcher。

## 8. 常见问题

### 问题 1：docker compose watch 报 .env 不存在

处理：

```bash
cp .env-example .env
```

然后重新执行：

```bash
docker compose watch
```

### 问题 2：5432 端口被占用

当前仓库已经将数据库宿主机端口改为 15432。

如果你看到旧的 5432 绑定报错，通常是因为：

- 你启动的是旧容器
- 或者 compose 配置还没重新读取

处理：

```bash
docker compose down --remove-orphans
docker compose watch
```

### 问题 3：服务创建了，但不知道算不算成功

不要只看 `Created`。

执行：

```bash
docker compose ps -a
curl http://localhost:8000/api/v1/utils/health-check/
```

只有 backend healthy、db healthy、health-check 返回 true，才能算真正跑通。

## 9. 建议的本地启动顺序

以后建议固定按这个顺序：

```bash
cp .env-example .env
docker compose watch
```

如果你怀疑是旧容器或旧网络影响：

```bash
docker compose down --remove-orphans
docker compose watch
```
